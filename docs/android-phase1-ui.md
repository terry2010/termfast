# TermFast Android — Phase 1 界面设计

> 基于 `docs/android-implementation-plan.md` §4.1.3 前端规划 + 桌面端 `src/components/shared/*` 功能映射。
> **Phase 1 不含交互式 SSH 终端**（终端是 Phase 2 范围）。

## 1. 设计约束

| 约束 | 说明 |
|---|---|
| UI 路线 | 原生 Kotlin + Jetpack Compose（Phase 0 决策推荐） |
| 导航模式 | 底部 Tab（服务器 / 日志 / 设置） |
| 最小宽度 | 360dp（覆盖主流手机） |
| 主题 | Material 3，支持 light/dark 跟随系统 |
| 无终端 | Phase 1 不做交互式 SSH 终端，服务器详情无"终端"Tab |
| `connected` 语义 | Android Phase 1 的 `connected` = SSH 连接成功，**不再等待终端就绪**（桌面端 `ServerList.tsx` 把 `connected` 耦合到终端打开，FFI 层需解耦：Android 侧 SSH 连接成功即发 `connected`，不依赖终端会话） |
| 无系统代理 | Android 无桌面端"系统代理"概念，移除该功能 |
| 无自动更新 | 走 Google Play，移除"检查更新"入口 |

## 2. Phase 1 功能范围

### 2.1 保留（从桌面端移植）

| 桌面端组件 | Android 对应 | Phase 1a | Phase 1b |
|---|---|---|---|
| ServerList | 服务器列表页 | ✅ | ✅ |
| AddServerDialog | 添加/编辑服务器页 | ✅ | ✅ |
| ServerDetail（概览/代理/触发器） | 服务器详情页（Tab：概览/代理/触发器） | ✅ 概览+代理 | ✅ +触发器 |
| LogViewer | 日志页 | ✅ | ✅ |
| SettingsPage（通用/日志/通知/关于） | 设置页 | ✅ 子集 | ✅ |
| VPN 开关 | 首页顶部 + 快捷通知 | ✅ | ✅ |
| per-app 代理 | 设置页 → per-app 白名单 | ❌ | ✅ |
| Kill switch | 设置页 → kill switch 开关 | ❌ | ✅ |

### 2.2 移除（不适用于 Android 或 Phase 2+）

| 桌面端功能 | 移除原因 |
|---|---|
| TerminalView | Phase 2 范围 |
| 系统代理设置 | Android 无此概念 |
| 自动更新 / 检查更新 | 走 Google Play |
| 最小化到托盘 | 移动端无托盘 |
| 窗口标题栏 | 移动端无自定义标题栏 |
| 拖拽排序服务器 | 移动端用长按拖拽，Phase 1a 暂不做 |
| 键盘快捷键 | 移动端无键盘 |

## 4. 状态机与 VPN 开关交互

### 4.1 VPN 开关状态流转

```
                    ┌──────────┐
                    │  idle    │  未连接 SSH，VPN 开关禁用
                    └────┬─────┘
                         │ 点击"连接"
                         ▼
                    ┌──────────┐
              ┌─────│connecting│  SSH 连接中
              │     └────┬─────┘
              │          │ SSH 连接成功（connected = SSH 就绪，不等终端）
              │          ▼
              │     ┌──────────┐
              │     │ connected│  SSH 已连接，VPN 可开启
              │     │ no-vpn   │
              │     └────┬─────┘
              │          │ 点击"开启 VPN"
              │          ▼
              │     ┌──────────┐  VpnService.establish() + tun2proxy
              │     │ connected│
              │     │ vpn-on   │◀──────────────┐
              │     └────┬─────┘               │
              │          │                     │
              │     ┌────┴──────────────┐      │
              │     │                   │      │
              │     ▼                   ▼      │
              │  点击"关闭 VPN"     onRevoke    │
              │     │              (系统撤销)   │
              │     ▼                   │      │
              │  ┌──────────┐            │      │
              │  │ connected│            │      │  SSH 保持，仅 VPN 关
              │  │ vpn-off  │────────────┘      │
              │  └──────────┘                   │
              │     │                           │
              │     │ 点击"开启 VPN"            │
              │     └───────────────────────────┘
              │
              │ SSH 断开 / 认证失败 / 网络断开
              ▼
         ┌──────────┐
         │reconnect-│  auto_reconnect 开启时：指数退避重连
         │ ing      │  ── 成功 ──▶ connected (no-vpn 或 vpn-on，按断开前状态恢复)
         └────┬─────┘  ── 失败/超时 ──▶ disconnected
              │
              │ 关闭连接 / 重试耗尽
              ▼
         ┌──────────┐
         │disconnected│
         └──────────┘
```

**与旧版差异**：
- **网络断开不再转到 `connected+vpn-off`**：网络断开会同时断 SSH，应进入 `reconnecting`（auto_reconnect 开启）或 `disconnected`。
- **`onRevoke` 单独画分支**：`onRevoke` 是用户在系统层面撤销 VPN 授权，SSH 保持，转 `connected+vpn-off` 并同步 app 内开关为关；与"用户点关闭 VPN"语义不同但终态相同。
- **`connected` 语义**：SSH 连接成功即 `connected`，不等终端（见 §1 设计约束）。

**Kill switch 状态机（Phase 1b，叠加在上述状态上）**：
```
connected + vpn-on  ──VPN 失败/断开──▶  kill-switch-on: 空 TUN 黑洞（establish 但不跑 tun2proxy）
                                        │
                                        ├── SSH 重连成功 ──▶ connected + vpn-on
                                        └── 用户关闭 VPN ──▶ off（不建立 TUN）
```
> Kill switch 开启时，VPN 断开期间不回退到真实网络，而是建立"空 TUN"让流量落入黑洞，避免泄漏（见 implementation plan §4.1.2）。

### 4.2 卡片状态显示规范

| 状态 | 圆点 | 颜色 | 卡片副标题 | VPN 开关 |
|---|---|---|---|---|
| connected + vpn-on | ● 实心圆 | 绿 #22c55e | 出口IP · 延迟 | 开启 |
| connected + vpn-off | ● 实心圆 | 绿 #22c55e | 出口IP · 延迟 | 关闭 |
| connecting | ◐ 右半圆 | 黄 #eab308 | 连接中... | 禁用 |
| reconnecting | ◑ 左半圆 | 黄 #ca8a04 | 重连中... | 禁用 |
| auth_failed | ■ 方块 | 红 #ef4444 | 认证失败 | 禁用 |
| disconnected | ○ 空心圆 | 灰 #9ca3af | 已断开 | 禁用 |
| offline | ○ 空心圆 | 灰 #6b7280 | 离线 | 禁用 |

## 5. 数据流（JNI 事件 → Compose UI）

```
Rust crates/android-ffi                Kotlin                    Compose UI
┌──────────────┐          ┌────────────────────┐          ┌──────────────┐
│ ServerManager│          │ RustBridge         │          │ ViewModel    │
│ ProxyServer  │──event──▶│ (JNI callback)     │──Flow───▶│ (StateFlow)  │
│ tun2proxy    │  JSON    │ → emit to Flow     │          │ → State      │
└──────────────┘          └────────────────────┘          └──────┬───────┘
                                                                │ collectAsState
                                                                ▼
                                                          ┌──────────────┐
                                                          │ Composable   │
                                                          │ (重组渲染)    │
                                                          └──────────────┘
```

**事件 JSON schema（与桌面端 IPC 事件对齐；事件机制与回调方式见 implementation plan §4.1.1 `subscribe_events`）**：

`server:status_changed`：
```json
{
  "type": "server:status_changed",
  "server_id": "uuid",
  "status": "connected",
  "exit_ip": "1.2.3.4",
  "latency_ms": 45
}
```

`proxy:status_changed`：
```json
{
  "type": "proxy:status_changed",
  "server_id": "uuid",
  "proxy_running": true,
  "active_channels": 5
}
```

`log:entry`：
```json
{
  "type": "log:entry",
  "entry": {
    "id": "uuid",
    "timestamp": "2025-01-15T14:30:15Z",
    "server_id": "uuid",
    "level": "info",
    "category": "Connection",
    "message": "SSH 已连接到 1.2.3.4"
  }
}
```

> 其他事件类型（如 `vpn:status_changed`、`trigger:executed`）的 schema 待 implementation plan §4.1.1 补全后对齐。
